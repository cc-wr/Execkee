# Execkee — Status

_Last updated: 2026-06-20._

A controller-and-workhorse system that manages permanent Claude Code instances,
runs a 30-minute life-tracking cycle, and surfaces the single most pressing issue
on a live dashboard answered in natural language via a primary Claude Code surface.
See [`claude-life-manager-architecture.md`](./claude-life-manager-architecture.md)
for the full design and its codicils.

## Phase status

- **Phase 0 (single machine):** complete — create/adopt/hide/foreground/close, fork-reports with model fallback, 30-min cycle → dashboard sentence, in-instance hide/close hook.
- **Phase 1 (multi-machine):**
  - ✅ Workhorse decoupled from shared filesystem — instance **state** syncs over the socket (`SYNC` / `STATE_UPDATE`, master tracking on the controller).
  - ✅ **Session discovery + adoption routing across machines**: `sessions` is aggregated from every connected workhorse and **grouped by host**; `manage` auto-routes to the workhorse that owns the session (or `--on <workhorse-id>`); `create` takes `--on` to target a machine.
  - ✅ **Un-adopt / release**: `unmanage <id>` stops managing an instance (leaves its window running) and removes it from master tracking; the primary understands "release / un-adopt".
  - ✅ Controller is **brain-only by default** (workers on remote machines); set `EXECKEE_LOCAL_WORKHORSE=1` / `-WithLocalWorkhorse` for a co-located workhorse (single-machine). Stale `wh-local` records are purged from tracking on brain-only startup.
  - ✅ Bootstrap installs a real **git clone** (committable) and installs Git for Windows portably if missing; on re-run it **auto-`git pull --ff-only`**.
  - ✅ **Resilience — dead-workhorse detection**: the hub drops a connection silent for ~3 heartbeat intervals (~90s), not only on a clean socket close.
  - ✅ **Boot-persistence (logon)**: `scripts/install-startup.ps1` registers a no-admin Startup-folder launcher so the controller/workhorse starts at logon.
  - ✅ **Primary resume-on-restart** (KI-4): the supervisor captures the primary's session id from the life-tasks project dir and `--resume`s it on restart, falling back to a fresh launch on any miss (`launchPrimaryWindow`).
  - ✅ **Created-instance session capture** (KI-1): a session-less created instance now polls for the new transcript it writes, matches it by cwd, and adopts its session id locally; the controller folds it into master tracking first-write-wins — so crash-recovery / restart resume *with* context instead of starting fresh.
  - ✅ **Bidirectional settings sync** (`settings-sync.js`): the controller is the canonical store + rebroadcaster over the existing WS hub; each node watches its own `~/.claude` and reports genuine local edits; last-write-wins by mtime; loop-guarded by content hash. Synced files are an **explicit allowlist** — `settings.json` and the global `CLAUDE.md` only — so credentials/state (`.credentials.json`, `.claude.json`, `sessions/`, `history.jsonl`, …) cannot leak by construction. Off-switch: `EXECKEE_SETTINGS_SYNC=0`.
  - ✅ **Native Remote Control** (**on by default**; opt out with `EXECKEE_REMOTE_CONTROL=0`): managed instances + the primary launch with Claude Code's `--remote-control`, so each window is drivable from claude.ai/code or the mobile app. Verified RC works on the dev machine (v2.1.178, claude.ai auth); empirically confirmed `--remote-control` **degrades gracefully** (an RC-ineligible launch still starts a normal session — does NOT crash), so no detect-failure/fallback path is needed. Requires claude.ai OAuth (not API key) + Pro/Max; caveat: a >10 min network outage exits the `claude` process (Execkee then relaunches it).
  - ✅ **Daily plan with tentative guesses** (`daily-plan.json`): the task list **resets each local day** — completed items archive, incomplete confirmed ones carry forward, and Execkee **guesses a full horizon of tasks from the tracked files** (TRACKING.md + context-sources, once/day), tagging which are achievable **today**. Guesses are **tentative (LLM-generated)** until the user **approves via the primary** (`approve-task`/`reject-task`; approval promotes into `tasks.json`). "Your Tasks" shows backlog + **today's tentative subset**; the **"Tracked · no instance"** panel shows the **rest of the guessed horizon** with no managed instance (matched by name) — the two are disjoint (no repeats). Brief v11: any completion (tentative or confirmed) prompts the primary to update the tracked source docs (e.g. a .docx) / TRACKING.md. Tentative guesses carry an LLM-estimated **priority** chip and are ordered by a **weighted-random** sort (priority-biased; re-rolled each ~30-min cycle so a low-priority guess can occasionally reach the top).
  - ✅ **Structured deferrals** (`deferrals.json`): the primary records a hold with `defer "<topic>" [--until YYYY-MM-DD]` (CLI/API), and the cycle suppresses that topic's related **guessed tasks** + instance-surfaced (presumed) tasks + sentences from the dashboard (deferring works on a guessed task too, not just instance topics). Relatedness is judged by an **LLM relatedness check** (by meaning — catches paraphrases, avoids coincidental word-share false positives), with a text match only as a fallback if the model is unavailable; active-by-date. Scheduled guesses are covered **airtight** via a persisted per-entry `deferred` flag (maintained by the LLM on defer/undefer and each cycle), so a deferred scheduled guess never re-injects on a sync rebuild and returns when the deferral lifts/expires. `undefer`/`deferrals` manage them. Replaces relying on freeform TRACKING.md prose; brief v13 directs the primary to use `defer`.
  - ✅ **Schedule an approved task for later**: a backlog task in `tasks.json` with a `startDate` (YYYY-MM-DD) stays hidden from the plan until that date, then appears as a normal **confirmed** task (no approval). The primary sets the field directly (brief v16). Distinct from `schedule-guess` (which queues a *tentative* guess to approve).
  - ✅ **Scheduled guesses** (`scheduled-guesses.json`): the mirror of a deferral — `schedule-guess "<text>" --on YYYY-MM-DD [--until …] [--horizon]` (CLI/API) surfaces a user-specified task as a **tentative guess** in the plan on/after its start date, then it flows through the normal approve/reject (which clears the schedule). `--horizon` routes it to "Tracked · no instance"; `unschedule-guess`/`scheduled-guesses` manage them. Brief v14 maps "next week, have X as a guessed task" → the primary computes the date and runs `schedule-guess`.
  - ✅ **macOS workhorse — feature parity** (`adapter-mac.js`): the workhorse's ONLY OS-specific surface, the window/process adapter, now sits behind a dispatcher (`src/workhorse/adapter.js`, dynamic-import by `process.platform`). `adapter-win.js` is **unchanged**; `adapter-mac.js` drives **Terminal.app via AppleScript/osascript** — each instance is a Claude CLI in its own window: launch via `do script` (new window, returns a tab → cache the window id), hide/show via the window's `visible` + `activate`/`frontmost`, **exact PID capture** via a per-instance launcher that does `echo $$ >pidfile; exec claude …` (exec preserves the pid), and kill = SIGTERM/SIGKILL the process group + close the window by matching the tab's `tty`. Installable via bash mirroring the PowerShell flow: **`bootstrap.sh`** (portable Node tarball + `claude.ai/install.sh`, no admin), **`execkee-workhorse.sh`** (launcher), **`scripts/install-startup.sh`** (a login **LaunchAgent**). The only edits to existing files are one import line each in `instances.js`/`instance-hook.js` — verified to leave Windows byte-identical (the dispatcher loads `adapter-win.js` on win32 with all 7 functions; `git diff` on `adapter-win.js` is empty). **Caveats (see KI-8):** driving Terminal needs a one-time macOS **Automation (TCC)** approval at the console; and the AppleScript/`osascript` path is written from documented semantics but **not yet exercised on a physical Mac** — it needs a live shakedown.
  - ✅ **Probe reports** (**on by default**; opt out with `EXECKEE_PROBE_REPORTS=0`): generate a status report by driving the **live instance window** instead of forking its on-disk transcript — which is stale for Remote-Control-bridged sessions (KI-9). The workhorse reads the instance's console **frame**, and only when it is **idle** (status bar up, not generating) and its state **changed since the last probe** (a stored end-marker) does it **inject** a short prompt asking the model to print a marker-delimited status report, then reads that report back from the frame (decoration glyphs stripped). Transcript-agnostic — works wherever turns are persisted. It appends a benign probe turn to the user's live conversation when idle, and falls back to the fork report whenever the window doesn't behave as expected. **Windows** (`probe-win.js`): `AttachConsole` + `ReadConsoleOutputCharacter` / `WriteConsoleInput` — **validated live** (an injected prompt was answered and a clean report parsed; idle-detection + Enter-submit + marker extraction all confirmed across repeated trials). **macOS** (`probe-mac.js`): AppleScript — `get contents of <tab>` to read, `do script "<text>" in <tab>` to inject into the running claude on that tab (mapped pid→tty→Terminal tab); **implemented, needs Mac-hardware verification** (KI-8). **On by default** (opt out with `EXECKEE_PROBE_REPORTS=0`); it appends a probe turn to the user's real conversation, so it is guarded to never inject mid-inference, never at a permission/trust prompt, never re-probe an unchanged session, and the prompt instructs the model to change no files. On any hard probe failure it **falls back to the existing fork report**. Orchestration + platform dispatch in `src/workhorse/probe.js`.
  - ✅ **Process + chat logging** (`src/common/logger.js`): the server (`controller.log`), workhorse (`workhorse.log`), and supervisor (`supervisor.log`) each tee their full stdout/stderr to a rotating file under `~/.execkee/logs/`, and the primary's conversation is tailed to `primary-chat.log` (readable USER/ASSISTANT turns; attaches at the current end so no backlog dump; freezes if the primary is RC-bridged, per KI-9). So when a bug is raised the logs are at a fixed, known path — inspect with `node src/cli.js logs [name] [--tail N]` or read the files directly. Implementation: `initProcessLog(name)` monkey-patches stdout/stderr write to also append (original write still hits the terminal, so behavior is unchanged), size-capped with one rollover (`<name>.1.log`), using `appendFileSync` so rotation's rename works on Windows even with the file in use. Fully guarded (try/catch everywhere); off-switch `EXECKEE_LOG=off`, cap `EXECKEE_LOG_MAX_BYTES`.
  - ⏳ Remaining: **before-logon service** (vs the logon-startup above); **resume reverts to stale state** (KI-6, below); **live macOS shakedown** (KI-8, now also covering the macOS probe).
  - ⏭️ Deferred: Linux workhorses (the mac adapter is a ready template — swap Terminal/AppleScript for the local terminal emulator); a code-signed app bundle + PPPC profile to pre-approve TCC Automation (needs admin/MDM).

## Known issues / future work

- **KI-1 — created-instance session capture (RESOLVED 2026-06-16).** A `create`d instance starts with no `sessionId`; on recovery it relaunched *fresh*, losing all conversation history accumulated since creation (the confirmed root cause of "recovery loses history" for created instances — seen live as the `inst-dtest` record with `sessionId: null`). **Fixed (commit `69e95f1`):** `createInstance` snapshots known session ids before launch, then `_captureCreatedSession` polls (fire-and-forget, ~24 s) for the new transcript, identifies it by cwd match, and stores its id locally via `updateInstance`; the next `STATE_UPDATE` carries it to the controller, which folds it first-write-wins (`hub.js`). The earlier crash-loop guard (session-less → fresh window via `createNewInstance`; helper drops empty argv tokens) remains as the safety net for instances whose capture misses.
- **KI-2 — cross-machine adoption: id-resolution fixed; full path still to confirm.** A truncated session id (the primary adopting `de9066ff` instead of the full id) caused "Session not found on disk" even for a listed session. Adoption now resolves a unique prefix to the full id (`resolveSessionId`). A complete cross-machine adoption (launch + first report on a remote workhorse) still needs a live end-to-end run, and is gated by KI-1.
- **KI-5 — closing the server doesn't close the primary surface (deferred).** Ctrl+C on the controller window triggers a shutdown that taskkills the primary, but killing only the server process (which the supervisor then restarts) leaves the primary open. Deferred — the server is slated to become a Windows service, which reshapes lifecycle/teardown anyway.
- **KI-3 — boot persistence: logon-startup added; before-logon pending.** `scripts/install-startup.ps1` auto-starts Execkee at logon (no admin). A before-logon Windows **service / scheduled task** (runs without an interactive session) is still pending.
- **KI-6 — resume reverted to stale conversation state (FIX IMPLEMENTED 2026-06-16; needs live verification).** Reported: the brain resuming the **primary surface** keeps coming back to the same old state; **workhorse** instances had the identical problem. **Corrected diagnosis** (via Claude Code docs): `claude --resume <id>` *continues* the same id in both interactive and headless mode — it does NOT auto-fork. So staleness = the *stored* id isn't the *live* one: (1) the primary's `resumablePrimarySession()` returned null (capture miss / cwd-gate / missing file) → **fresh seeded relaunch every restart**; (2) the hub folded `sessionId` first-write-wins → a live-id change was ignored and a stale id re-synced. **Fix:** (a) the in-instance hook (`instance-hook.js`) captures the exact live `session_id` on every prompt and updates the stored id on change (unit-verified); (b) the hub now accepts any non-empty `sessionId` change; (c) the supervisor refreshes `primary-session.json` to the newest life-tasks session on a 15 s timer and `resumablePrimarySession()` falls back to it instead of a fresh seed. Couldn't reproduce locally (the 9 live workhorse instances all had stored==live; the primary runs on the controller). Still needs a live restart test on the new code. **2026-06-20:** also fixed the *report* path — `report()` now adopts the **newest session in the instance's project dir** (`newestSessionInSlugOf`) before the change-check + fork, so a status report summarizes the LATEST conversation even if it continued/forked into a new id (diagnosed real drift: "Main Execkee" stored `a220b829` while the live session was `c8e9cdbc`). **Problem-1 hardening (same day):** a `SessionStart` hook (alongside `UserPromptSubmit`) now captures the instance's own live `session_id` immediately on launch/resume/compact, so `inst.sessionId` stays exact without dir-guessing; and the report's newest-in-slug fallback **never adopts a session claimed by another managed instance** (no stealing a sibling's conversation in a shared dir). Residual: an unrelated *unmanaged* conversation in the same dir could still be picked — re-adopt it to make tracking exact. **No-re-adopt live-transcript fix (2026-06-20):** the in-instance hook now records the **exact `transcript_path`** Claude passes on every prompt (verified the running instances have `EXECKEE_INSTANCE_ID` + correct data dir in their env, so the existing hook fires), and `report()` reads that file directly (size for change-detection, fork by the live id) — so reports follow the live conversation even if it forked/moved to another id or dir, with **no re-adopt**. Takes effect after a `git pull` in `~/Execkee` (hook file is re-read per prompt) + a workhorse restart (to load the new `report()`); instances are NOT relaunched. Diagnostic note: the all-skip seen in log1w was because the instances were idle since the 02:19 deploy (transcripts last flushed at launch), not a logic failure.

- **KI-4 — resilience: dead-detection + primary-resume both shipped.** The hub drops silent/dead workhorse connections via heartbeat (~90s). **Primary resume-on-restart** is implemented (commit `a55c633`): `launchPrimaryWindow` captures the primary's session id from the life-tasks project dir and `--resume`s it across restarts, falling back to a fresh seeded launch on any miss.

- **KI-7 — large / active transcripts are unreportable by the fork method (OPEN; hard problem; logged 2026-06-20).** Two compounding causes, both outside report-code correctness:
  - **(1) Size ceiling.** The cycle reports by forking `claude -p --resume <sessionId> --fork-session --no-session-persistence`, which loads the *entire* transcript into the model's context window. A long-lived window's `.jsonl` grows to hundreds of MB (live counterexample: the **"Current Words"** session `8b408bf7…` at **126 MB**), which **exceeds the context window** — so even when the file *has* changed, a fork-report cannot ingest it. The biggest, most-worked windows are precisely the ones that can't be summarized.
  - **(2) On-disk lag vs the live window.** The on-disk transcript can trail the live conversation (a window actively in use can show a `.jsonl` whose `size == watermark` / `mtime == launch time`), so change-detection sees no growth and skips. (Per KI-6's findings the common case is genuine idleness, but for a *busy* window the file can still lag what's on screen.)
  - **Candidate approaches the user named (not yet chosen / built):**
    - **(a) Close-to-flush** — close the window so Claude finalizes the transcript file, then report from the now-complete `.jsonl`. Cost: disrupts the live session.
    - **(b) Compact-for-reports** — compact the transcript (or summarize in chunks / map-reduce over the file) so a big one fits the context before reporting.
    - **(c) In-window report + rewind UI** — generate the report *inside* the live window (native in-session summarization) and use the rewind UI, instead of an external fork that must reload everything.
  - Also previously floated by Execkee (a fourth option, not on the user's list): **delta reporting** — read only the transcript bytes *since the watermark* and summarize the delta with `claude -p` (no `--resume`), sidestepping both the size ceiling and the reload. **Decision pending — do not implement until the user picks an approach.**

- **KI-8 — macOS workhorse needs a live shakedown + a one-time TCC approval (OPEN; logged 2026-06-20).** The macOS workhorse (`adapter-mac.js` + `bootstrap.sh` / `execkee-workhorse.sh` / `scripts/install-startup.sh`) was implemented and statically verified on Windows (JS `node --check`, `bash -n`, the dispatcher loads `adapter-win.js` on win32 with all 7 functions, the generated helper/launcher pass `bash -n`, `set -e` edge cases tested), and its AppleScript/`osascript` and install facts were researched against Apple-aligned sources. **But it has not been run on a physical Mac** — so the AppleScript semantics (`do script` → tab → window id; `visible`/`frontmost`; `tab.tty` = `/dev/ttysNNN`; `close … saving no`), the `ps -o tty=`/`pgid=` formats, and the portable-Node/`install.sh`/LaunchAgent flow still need a live confirmation. **Operational caveat:** controlling Terminal.app via AppleScript triggers a **one-time macOS Automation (TCC) consent** ("… wants to control Terminal.app") that a human must approve **at the console** — a launchd/headless first run fails silently until then. The grant is bound to the `node` binary path, so the installer pins the stable `~/.execkee-tools/node/current` symlink; do the first run interactively at the screen, then install the LaunchAgent. Pre-approving without a prompt needs a signed app bundle + a PPPC profile (admin/MDM), which is out of scope for the no-admin install. Suggested shakedown: on a Mac, run `bootstrap.sh --controller <ip>:7700`, approve the prompt, then `manage`/`foreground`/`hide`/`close` one instance and confirm a report flows to the dashboard. **Reviewed (14-agent adversarial pass, 2026-06-20)** — 6 confirmed mac-only findings fixed: pipe-delimited launch output (an empty pid no longer shifts the window id into the pid field) + orphan-window close on a failed launch; dropped the `front window` fallback that could hide the wrong window; strict-allowlist validation of `instanceId` and numeric validation of window-id/pid before they reach any `osascript` heredoc (closes AppleScript-injection vectors over the controller→workhorse link); and a guarded `launchctl bootstrap`. **Remaining documented parity gap:** Windows removes the window's `SC_CLOSE` menu item so the X can't kill an instance; Terminal.app has no scriptable equivalent, so a user *can* close an instance window — but Terminal's default "prompt before closing a running process" softens it and the subcontroller relaunches the closed instance as a crash-recovery (permanence holds up to `CRASH_RETRY_MAX`). **Note:** the controller↔workhorse WebSocket remains unauthenticated (pre-existing; relies on the Tailscale-private network) — the adapter input-validation above is the scoped mitigation; full WS auth is a separate future item.

- **KI-9 — ROOT CAUSE of the stale-reports / "inputs to managed instances aren't written to file" problem: Remote Control bridging diverts the transcript OFF the local file (FOUND 2026-06-20, high confidence, reproduced on this dev session).** Supersedes the earlier "instances are just idle resumes" reading for the RC case. Execkee launches managed instances (and the primary) with `--remote-control` (default-on). When an instance is actually **bridged** — a remote client (claude.ai/code, the mobile app, or the local Claude **desktop app**, which is running here and can auto-attach) connects to it — Claude Code streams the conversation through the cloud **bridge** (local file gets `{"type":"bridge-session","bridgeSessionId":"cse_…"}` bookkeeping events) and **stops appending real turns to `~/.claude/projects/<slug>/<id>.jsonl`**. The local file freezes at the bridge point, so the cycle's fork-report sees `size == watermark` and skips — for every bridged instance, indefinitely.
  - **Evidence (read-only inspection of the live workhorse):** for all 12 managed instances the *real* end-of-file (read through an open handle, so not a stale-metadata artifact — `delta=0`) equals the stored watermark exactly. RC-bridged sessions (`a220b829` = this very session; `45dc12df`) **end on a `bridge-session` event with no real turn after it**, yet their conversations demonstrably continued: this session's local file has its last real turn at **2026-06-16T09:23Z** and contains **zero** of today's work (`adapter-mac`, `feature parity`, `Armenia` → 0 matches) even though the session is live. The **primary** (`76047165`), driven *locally* rather than through the bridge, keeps writing turns normally (last turn today). Non-bridged instances are genuinely idle. Live `claude.exe` command lines confirm every managed instance carries `--settings instance-settings.json`, and the bridged ones carry `--remote-control`.
  - **Corroboration:** Claude Code docs + issues #34581 / #57361 — the local `.jsonl` freezing during Remote Control is a known limitation; `cse_` is short-lived **routing**, not durable storage; there is **no documented flag to keep local persistence on, and no documented way to fetch the bridged transcript back** (window close is not documented to flush it → possible data-durability risk for bridged work, flagged, medium confidence).
  - **Key implication:** Remote Control and local-transcript-based reporting are **mutually exclusive for the same instance** under current Claude Code behavior. Every file-based report path (the fork, delta-reads, SDK `getSessionMessages`) fails for a bridged instance because the source file is frozen.
  - **Fix options (decision pending — NOT acted on per the request):**
    1. **Don't bridge managed instances** — drop `--remote-control` from the *instance* launch (keep it for the primary if desired). Restores local persistence so the existing fork-report works again (still subject to the KI-7 size ceiling). Cleanest + highest-confidence, but sacrifices driving instances from mobile/web.
    2. **In-window / "probe" reporting — IMPLEMENTED (on by default, opt out `EXECKEE_PROBE_REPORTS=0`; see the "Probe reports" feature above).** Generates the report by driving the live instance window (read the console/Terminal frame, inject a marker-delimited report prompt, read the reply) — transcript-agnostic, so it works for bridged sessions and for the KI-7 oversized-transcript case alike. Done via OS console/Terminal automation (not the RC bridge protocol): Windows `AttachConsole`+`WriteConsoleInput`/`ReadConsoleOutputCharacter` (validated live), macOS AppleScript `do script … in <tab>`/`get contents` (pending hardware test). It still appends a probe turn to the user's real conversation — hence opt-in, idle-guarded, change-nothing prompt, and a fork-report fallback.
    3. **Detect-and-surface** — detect a bridged instance (local file ends on `bridge-session`, not growing) and label it "live via Remote Control — not locally reportable" instead of silently skipping. Cheap honesty pending a real fix.
  - Full options-vs-probe comparison is in the 2026-06-20 findings email.

## Bug review — "session finding" + "recovery loses history" (2026-06-16)

A rigorous review of two reported bugs. Method: empirical (live process/transcript
inspection + a headless and an interactive `--resume` test), no destructive changes.

**Bug A — "can't find sessions where the tokens aren't loaded."** Not a filtering
bug. `listLocalSessions` enumerates *every* `.jsonl` one level under each
`~/.claude/projects/<slug>` with no size/content/"token" filter; subagent
transcripts (nested under `subagents/`) are correctly excluded. The real cause is
**topology/connectivity**: `hub.listSessions()` fans out only to *connected*
workhorses, so (a) a session on the brain-only controller's own machine is never
enumerated (no local workhorse to ask), and (b) a workhorse that is momentarily
disconnected (heartbeat reconnect) yields an empty list. The CLI already prints
"No workhorses connected" / per-host "(could not list: …)". *Recommended follow-up
(not yet done):* also surface the controller-local sessions, and a per-host count.
**Update (2026-06-16):** found a concrete instance — a session you adopted then
**closed** stayed tagged `managed` and was filtered out of the default `sessions`
list *forever*, so the primary "couldn't find" it for re-adoption (hit when trying
to adopt this dev session). Fixed: only ALIVE/CLOSING instances occupy their session;
closed/failed ones release it back to adoptable (`hub.listSessions`).

**Bug B — "recovery loses history since initial adopt."** Two distinct paths:
  - **Created instances (root cause, FIXED):** a created instance had no `sessionId`,
    so crash/restart recovery relaunched it as a *fresh* session — total history loss.
    Confirmed live (`inst-dtest`, `sessionId: null`). Fixed by KI-1 (`69e95f1`).
  - **Adopted instances (recovery is correct):** `_launchAndMonitor` relaunches with
    the stored `sessionId`, the session's own cwd (`projectPath` = `getSessionCwd`),
